# Example Issue Template

Use this only when the repo does not already provide an issue template.

If the repo contains an issue template, prefer the repo template first and adapt the wording to match the author's style.

This file is fallback guidance, not a mandatory structure.

## Generic Issue

## Summary
- One or two brief bullets describing the problem, gap, or requested outcome

## Context
- Why this matters now
- Any relevant constraints, links, or prior discussion

## Scope
- What should happen
- What is explicitly out of scope if that matters

## Acceptance Notes
- A short list of conditions that would make this done

## Bug Report

## Summary
- One or two brief bullets describing the failing behavior

## Impact
- Who or what is affected

## Reproduction
1. Step one
2. Step two
3. Observe the failure

## Expected
- What should happen

## Actual
- What happens instead

## Context
- Logs, screenshots, environment details, or links only when they materially help

## Notes

- Remove sections that do not help
- Keep wording concise
- Follow the author's normal section naming when obvious
- Add related issue references only when relevant
